 Cat (05.04.2001)

New dialog flags identified
Each dialog option has 4 bytes of flags. The second least significant byte is the journal byte (the first sets other options, and the highest 2 seem to be unused).

The second LSB works in the following way (tested in BG2 only):

0x01 -- set this bit to 1 if you want to add a journal. If no other bits are set, it adds as an "info" journal.
0x02 -- this bit must be 0, else it screws up the dialog?
0x04 -- if this is set in conjunction with bit 1, the journal is added to the "quests" journal, instead of the info journal.
0x08 -- if this is set in conjunction with bit 1, the journal is removed from the "quests" journal.

Editing the race selection screen would be hard. The BG executables are HUGE HUGE HUGE. And even the moderate sized executables I used to edit (yes, I've done some .EXE editing) were hard to find code in, and they were at most 400 KB or so of code. At something like 15 to 20 times as large, editing code in the executables is the needle in the haystack approach, not to mention it being difficult to edit executable code anyway. Especially static linked stuff like the Infinity Engine games.


If I had the source, I could do it... but reverse engineering something this big would be titanic in proportions. Even if you identified and removed all the standard libraries that are linked in there, there's tons of source files worth of code in that, and if you decompile, you get code sans any identifiers (like variable names, class names, etc) and sans comments

James Schumacher

that even if you could decompile it successfully (best chance is in assembler) you could edit the *.exe and recompile it. But we all know the infinity engine uses DirectX, which in itself is pretty much COM. C/C++ code is hard enough by itself to manage. When you look at assembler code you'll be even more baffled if you have no clue what you are looking at. And if you were able to get some decent complete assembler source (which is pretty easy with a decent decompiler) you are basically screwed anyway because what Cat said is true. C++ code and other languages when you look at the code, you know what does what cause the programmer (if he's smart anyway) will have good naming conventions for his classes/functions. Add in COM to all this mess and you're just asking for trouble

dualclass.2da

I was able to give the ability to dual class from a monk, sorceror, or barbarian to another class, but not to those classes. Problem is, only the barbarian works; monks and sorcerors could dual but would crash as soon as you hit the skills button in dual class selection. Barbarians work just fine. Just add this line to the dualclas.2da:

BARBARIAN 0 1 1 1 1 0

Don't try to change it so a barbarian can dual to a fighter or ranger as a barbarian counts as a fighter for some reason. This is why he can dual...remember, a dual class must also be a viable multiclass; that's why monks and sorcerers can't dual (there is no fighter/sorcerer multiclass on the demi-humans selection screen). This may also be the solution. Try and create a thief/sorcerer multiclass first, then a human swashbuckler to sorcerer dual may be legal.
