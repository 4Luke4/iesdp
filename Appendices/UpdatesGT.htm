ITM files update :

Effects/.............

i was just trying to make a halberd that casted contagion on dragons upon hit and i figured out some more unknowns in the item maker(v 1.2.2)! this new information affects the effmaker(v1.1) as well! 

to view what i am currently talking about open up sw1h32.itm and go to the other info screen. go to the item effect number 1. "Use EFF File". now go to "Click to edit parameters."

you are now viewing what i am talking about. those two parameters are ids and ids reference.  those two in particular are 146 (dragon reference number) and 4 (the race ids). this is part of the scheme used to cause this sword to do double damage to dragons. this particular effect causes fixed/random damage to dragons while the other Use eff on the sword causes damage vs creature type (dragon type), but this effect has parameters of 0 (nothing reference) and 2 (object.ids). i dont understand why they separated these effects though. they both only work on dragons, but why couldn't they just make effect that did 2d8 instead of two effects that did 1d8?

to further support my hypothesis, i submit the daystar(sw1h31). it is the other weapon used to cause double damage. it has 4 effects (two of which are relevant here). one is a use eff with parameters 0 and 2, the other is a use eff with paramaters of 4 and 3. as with peridan (sw1h32) the use eff with 0 and 2 parameters is a damage vs creature type effect (undead type). the use eff with parameters 4 (undead reference) and 3 (general.ids) is a fixed/double damage effect. (on a side note bioware put 1d8 in this parameter, which in my opinion was an error as following the example set by peridan the correct entry should have been 1d8+2 in order to correctly double the damage to undead).

what i'm trying to say is that "Use EFF" has the same parameter types as "Damage vs. Create Type."

by Devarrah :

bah! i was all wrong... i think i have it figured out now:
offset 0x0b is the compressed color index (and it's not always 0, items.bam is a good example) and this is for certain

Now transparency is a bit more experimental. The transparent color is 0,255,0 but if the palette doesn't contain 0,255,0, index 0 becomes the transparent color by default
There's still two bams that i find a bit problematic but if you use the above rules you'll be guaranteed a proper looking bam and if you decode by these rules you'll display all but 2 bams correctly (at least in bg) (i'll examine the two bams perhaps tomorrow and see if i can't come up with the what's going on)
